# Service Portal System - Requirements Artifacts

**Key UI Components:**
- Header: "University IT Service Portal"
- Navigation menu: Home | Submit Ticket | Track Request | Help
- Welcome message with image/logo
- "Submit New Request" button
- Services Available section listing:
  - Password Reset
  - Software Installation
  - Network Support
  - Account Access
- Support Announcements section showing:
  - Scheduled maintenance Friday

- Navigate to ticket submission
- Navigate to request tracking
- Access help/FAQ
- View available services and announcements

## B. Functional Requirements

### Navigation and Access (FR-1 to FR-4)

**FR-1:** The system shall provide a persistent navigation menu on all screens containing links to Home, Submit Ticket, Track Request, and Help.

**FR-2:** The system shall display the portal title "University IT Service Portal" on all screens.

**FR-3:** The system shall allow users to return to the home page from any screen via the Home navigation link.

**FR-4:** The system shall provide access to help/FAQ content from any screen via the Help navigation link.

### Service Information Display (FR-5 to FR-6)

**FR-5:** The system shall display a list of available IT services on the home page including Password Reset, Software Installation, Network Support, and Account Access.

**FR-6:** The system shall display support announcements on the home page including scheduled maintenance notifications and service updates.

### Ticket Submission (FR-7 to FR-11)

**FR-7:** The system shall provide a form for users to submit new support requests requiring Name, Email, Issue Category, and Problem Description.

**FR-8:** The system shall provide a dropdown menu for Issue Category selection containing predefined service categories.

**FR-9:** The system shall allow users to clear all form fields via a Clear Form button before submission.

**FR-10:** The system shall generate a unique ticket ID upon successful submission of a support request.

**FR-11:** The system shall capture and store the submission date and time for each ticket.

### Ticket Confirmation (FR-12 to FR-14)

**FR-12:** The system shall display a confirmation page after successful ticket submission showing the ticket ID, category, submitter, status, submission date, and estimated response time.

**FR-13:** The system shall set the initial status of newly created tickets to "Open".

**FR-14:** The system shall provide an estimated response time for each submitted ticket based on the issue category.

### Ticket Tracking (FR-15 to FR-18)

**FR-15:** The system shall allow users to search for tickets by entering a ticket ID.

**FR-16:** The system shall display ticket information including ticket number, category, current status, submitter, last updated date, and technician notes when a valid ticket ID is searched.

**FR-17:** The system shall display the current status of tickets using values such as "Open", "In Progress", or other status indicators.

**FR-18:** The system shall display technician notes or updates associated with the ticket to inform users of progress.

### Help and Support (FR-19 to FR-20)

**FR-19:** The system shall provide a Help/FAQ page containing frequently asked questions and answers related to common support tasks.

**FR-20:** The system shall display contact information for IT support including email address and alternative contact methods on the Help page.

## D. Ambiguities, Missing Information, and Assumptions

### Ambiguities:
1. **Authentication:** No login screen is shown. It's unclear if users must authenticate or if the portal is publicly accessible.
2. **User roles:** The distinction between "Student" submitter and other user types (faculty, staff) is unclear.
3. **Ticket ID format:** Multiple formats shown (IT-2045, 2025-164) - standardization needed.
4. **Category list:** Complete list of issue categories not defined beyond examples shown.
5. **Status values:** Full set of possible ticket statuses not specified.

### Missing Information:
1. **Validation rules:** No field validation requirements specified (e.g., email format, required fields).
2. **Error handling:** No error messages or validation feedback shown.
3. **Notification system:** Whether users receive email confirmations or updates is not specified.
4. **Edit capability:** No ability shown to update or cancel submitted tickets.
5. **Attachment support:** No file upload capability shown for supporting documentation.
6. **Search alternatives:** Only ticket ID search shown; no search by date, category, or status.
7. **Response time calculation:** Logic for determining estimated response time not defined.
8. **Technician interface:** No screens shown for IT staff to manage/respond to tickets.

### Assumptions Made:
1. Users can access the portal without authentication (or authentication occurs before reaching Screen 1).
2. All form fields on the submission screen are required.
3. Ticket IDs are system-generated and unique.
4. Users can only track their own tickets (or any ticket if they have the ID).
5. The system automatically timestamps submissions.
6. Status updates are performed by IT staff through a separate interface.
7. FAQ content is static and managed by administrators.
8. The portal is read-only for end users after ticket submission (no editing).
9. Email notifications are sent upon ticket submission (referenced in confirmation but not shown).
10. The "Estimated Response Time" is informational only and not a service-level commitment.
