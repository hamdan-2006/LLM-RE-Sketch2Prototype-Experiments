## B. Functional Requirements

### Navigation and Access (FR-1 to FR-3)

**FR-1:** The system shall provide a persistent navigation menu on all screens containing links to Home, Submit Ticket, Track Request, and Help pages.

**FR-2:** The system shall allow users to navigate between any main sections (Home, Submit Ticket, Track Request, Help) from any screen without losing unsaved data warning.

**FR-3:** The system shall display the portal title "University IT Service Portal" on all screens.

### Service Information Display (FR-4 to FR-5)

**FR-4:** The system shall display a list of available IT services on the home page including Password Reset, Software Installation, Network Support, and Account Access.

**FR-5:** The system shall display support announcements on the home page including scheduled maintenance notifications and new service availability.

### Ticket Submission (FR-6 to FR-10)

**FR-6:** The system shall provide a ticket submission form requiring Name, Email, Issue Category, and Problem Description fields.

**FR-7:** The system shall provide a dropdown menu for Issue Category selection containing predefined service categories.

**FR-8:** The system shall validate that all required fields (Name, Email, Issue Category, Problem Description) are completed before allowing ticket submission.

**FR-9:** The system shall generate a unique Ticket ID upon successful ticket submission.

**FR-10:** The system shall provide a "Clear Form" function that resets all input fields on the submission form to empty state.

### Ticket Confirmation and Tracking (FR-11 to FR-16)

**FR-11:** The system shall display a confirmation screen after successful ticket submission showing the confirmation message "Your support request has been submitted."

**FR-12:** The system shall display ticket summary information including Ticket ID, Category, Submitted By, Status, Submitted Date, and Estimated Response Time on the confirmation screen.

**FR-13:** The system shall allow users to search for existing tickets by entering a Ticket ID.

**FR-14:** The system shall display ticket details including Ticket Number, Category, Status, Submitted By, Last Updated date, and Technician Notes when a valid Ticket ID is searched.

**FR-15:** The system shall display current ticket status using predefined status values (e.g., Open, In Progress, Closed).

**FR-16:** The system shall display technician notes or updates associated with a ticket on the tracking screen.

### Help and Support (FR-17 to FR-18)

**FR-17:** The system shall provide a Help/FAQ page displaying frequently asked questions with corresponding answers.

**FR-18:** The system shall display IT support contact information including email address and phone number on the Help page.

### User Actions (FR-19 to FR-20)

**FR-19:** The system shall provide a "Return Home" or "Return to Home" button on confirmation and tracking screens to navigate back to the dashboard.

**FR-20:** The system shall provide a "Track Request" button on the confirmation screen to navigate directly to the ticket tracking page.

## D. Ambiguities, Missing Information, and Assumptions

### Ambiguities:

1. **Authentication/Login:** No login screen is shown. It's unclear if users must authenticate before accessing the portal or if it's open access.

2. **User Roles:** The system mentions "Student" as a submitter, but it's unclear if there are different user roles (student, faculty, staff) with different permissions or service access.

3. **Ticket ID Format:** Multiple formats shown (IT-2045, 2025-164). The actual format and generation logic is ambiguous.

4. **Issue Category Options:** The dropdown is shown but specific category options are not fully defined beyond examples (Network Support).

5. **Estimated Response Time Calculation:** How the "24 Hours" estimate is determined is not specified.

6. **Search Functionality:** Whether search requires exact Ticket ID match or supports partial matching is unclear.

### Missing Information:

1. **Error Handling:** No screens show error states (invalid ticket ID, submission failure, validation errors).

2. **Email Notifications:** Whether users receive email confirmations or updates is not shown.

3. **Ticket Updates:** How users receive notifications about status changes or technician responses.

4. **File Attachments:** No indication if users can attach screenshots or files to support requests.

5. **Ticket History:** Whether users can view all their past tickets or only search one at a time.

6. **Edit/Cancel Functionality:** No ability shown to edit or cancel submitted tickets.

7. **Priority Levels:** No indication of ticket priority or urgency settings.

8. **Service Level Agreements:** Beyond "24 hours," no detailed SLA information is provided.

9. **Multi-language Support:** No indication of language options.

10. **Accessibility Features:** No accessibility considerations shown in sketches.

### Assumptions Made:

1. **Public Access Assumption:** Assuming the portal is accessible without login, or login occurs before reaching Screen 1.

2. **Single Ticket View:** Assuming users track one ticket at a time rather than viewing a list of all their tickets.

3. **Read-Only Tracking:** Assuming users can only view ticket status, not interact with or update tickets after submission.

4. **Static FAQ:** Assuming FAQ content is static and managed by administrators, not dynamically generated.

5. **Email Validation:** Assuming basic email format validation occurs on the submission form.

6. **Ticket ID Uniqueness:** Assuming the system ensures all generated Ticket IDs are unique.

7. **Status Workflow:** Assuming standard ticket lifecycle: Open → In Progress → Closed (though only Open and In Progress are shown).

8. **Business Hours:** Assuming "24 hours" response time refers to business hours, not calendar hours.

9. **Contact Methods:** Assuming email and phone are the only support contact methods (no chat, in-person).

10. **University Context:** Assuming this is for a university environment based on "University IT Service Portal" title and "Student" user reference.
