## B. Functional Requirements

### Navigation & Access
**FR-1:** The system shall provide a persistent navigation menu on all screens containing links to Home, Schedule Appointment, My Appointments, and Help sections.

**FR-2:** The system shall display the clinic branding (logo/infographic) on the home page to identify the Community Health Clinic portal.

**FR-3:** The system shall allow users to return to the home page from any screen via a "Return Home" or "Home" navigation link.

### Appointment Scheduling
**FR-4:** The system shall provide a form for scheduling new appointments that collects patient name, email address, phone number, appointment type, preferred doctor, appointment date, appointment time, and optional additional notes.

**FR-5:** The system shall provide dropdown selection for appointment type including at minimum: General Checkup, Pediatric Care, Vaccination Services, and Follow-up Visits.

**FR-6:** The system shall provide dropdown selection for preferred doctor based on available clinic physicians.

**FR-7:** The system shall provide a date picker interface for selecting appointment dates.

**FR-8:** The system shall provide dropdown selection for available appointment time slots.

**FR-9:** The system shall validate that all required fields (patient name, email, phone, appointment type, preferred doctor, date, and time) are completed before allowing form submission.

**FR-10:** The system shall provide a "Clear Form" function that resets all input fields on the appointment scheduling form.

### Appointment Confirmation & Management
**FR-11:** The system shall generate a unique confirmation number (format: HC-####) for each successfully scheduled appointment.

**FR-12:** The system shall display an appointment confirmation screen showing patient name, appointment type, doctor, date, time, confirmation number, and status immediately after successful scheduling.

**FR-13:** The system shall assign an initial status of "Confirmed" to newly scheduled appointments.

**FR-14:** The system shall maintain a list of all patient appointments accessible via the "My Appointments" section.

**FR-15:** The system shall display appointment information in the "My Appointments" list including confirmation number, doctor name, appointment date, and current status.

**FR-16:** The system shall allow users to select an appointment from the list to view detailed information including location and special notes.

**FR-17:** The system shall provide a "Cancel App" function that allows users to cancel appointments from the "My Appointments" screen.

**FR-18:** The system shall support appointment status values including at minimum: Confirmed, Pending, and Cancelled.

### Information & Support
**FR-19:** The system shall display current announcements on the home page including information about available services and clinic updates.

**FR-20:** The system shall provide a Help/FAQ section containing answers to frequently asked questions about scheduling, cancellation, and confirmation processes.

**FR-21:** The system shall display clinic contact information (email and phone) on the Help page for additional support inquiries.

## D. Ambiguities, Missing Information, and Assumptions

### Ambiguities
1. **User Authentication:** No login/registration screens are shown. It's unclear if the system requires user authentication or operates as an open portal.
2. **Doctor Availability:** The mechanism for determining which doctors and time slots are available is not specified.
3. **Appointment Cancellation Workflow:** The confirmation or consequences of canceling an appointment are not shown.
4. **Status Transitions:** The conditions under which appointment status changes from "Pending" to "Confirmed" are unclear.
5. **Form Validation Rules:** Specific validation rules (e.g., email format, phone number format, date range restrictions) are not defined.

### Missing Information
1. **Error Handling:** No screens show error states or validation failure messages.
2. **Edit Functionality:** No capability to modify/reschedule existing appointments is shown.
3. **Notification System:** How patients receive confirmation (email, SMS) beyond on-screen display is not specified.
4. **Multi-patient Support:** Whether a single user can manage appointments for multiple family members is unclear.
5. **Appointment History:** No indication of how past appointments are handled or archived.
6. **Time Zone Handling:** Not specified if the system handles multiple time zones.
7. **Accessibility Features:** No indication of accessibility considerations (screen readers, keyboard navigation, etc.).

### Assumptions Made
1. **Single User Session:** Assumed the system tracks appointments for a single patient/user session without explicit login.
2. **Real-time Availability:** Assumed the dropdowns for doctor and time reflect real-time availability.
3. **Immediate Confirmation:** Assumed appointments are immediately confirmed unless marked "Pending" for specific reasons (e.g., doctor approval needed).
4. **Sequential Workflow:** Assumed the typical user flow is: Home → Schedule → Confirmation → My Appointments.
5. **Unique Confirmation Numbers:** Assumed confirmation numbers are system-generated and unique across all appointments.
6. **Contact Information Validity:** Assumed the clinic contact information shown on the Help page is current and monitored.
7. **Cancellation Permissions:** Assumed patients can cancel their own appointments without requiring clinic approval.
8. **Date Format:** Assumed standard date format (MM/DD/YYYY) based on the date picker representation.
