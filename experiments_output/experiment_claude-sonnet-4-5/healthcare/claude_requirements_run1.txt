## B. Functional Requirements

### Navigation & Access (FR-1 to FR-3)

**FR-1:** The system shall provide a persistent navigation menu on all screens containing links to Home, Schedule Appointment, My Appointments, and Help sections.

**FR-2:** The system shall allow users to return to the home page from any screen via the navigation menu or dedicated "Return Home" buttons.

**FR-3:** The system shall display the clinic branding (logo/infographic) and portal title "Community Health Clinic Portal" on the home screen.

### Appointment Scheduling (FR-4 to FR-10)

**FR-4:** The system shall provide a "Schedule New Appt" button on the home screen that navigates users to the appointment scheduling form.

**FR-5:** The system shall require users to enter patient name, email address, and phone number when scheduling an appointment.

**FR-6:** The system shall provide dropdown menus for selecting appointment type, preferred doctor, and appointment time.

**FR-7:** The system shall provide a date picker interface for selecting the appointment date.

**FR-8:** The system shall allow users to enter optional additional notes in a text area field on the scheduling form.

**FR-9:** The system shall provide a "Clear Form" button that resets all fields on the appointment scheduling form.

**FR-10:** The system shall validate all required fields before allowing appointment submission via the "Confirm Appt" button.

### Appointment Confirmation (FR-11 to FR-13)

**FR-11:** The system shall generate a unique confirmation number (format: HC-####) upon successful appointment scheduling.

**FR-12:** The system shall display an appointment confirmation screen showing patient name, appointment type, doctor, date, time, confirmation number, and status after successful booking.

**FR-13:** The system shall display a success message "Your appointment has been scheduled successfully" on the confirmation screen.

### Appointment Management (FR-14 to FR-17)

**FR-14:** The system shall display a list of upcoming appointments showing confirmation number, doctor name, date, and status on the My Appointments screen.

**FR-15:** The system shall support at least two appointment statuses: "Confirmed" and "Pending".

**FR-16:** The system shall display detailed information for a selected appointment including location and special notes (e.g., "Arrive 15 minutes early").

**FR-17:** The system shall provide a "Cancel App" button on the My Appointments screen to allow users to cancel appointments.

### Information Display (FR-18 to FR-20)

**FR-18:** The system shall display available services on the home screen including General checkup, Pediatric care, Vaccination Services, and Follow-up Visits.

**FR-19:** The system shall display announcements on the home screen including time-sensitive information such as flu shot availability and form updates.

**FR-20:** The system shall provide a Help/FAQ page containing frequently asked questions about scheduling, cancellation, and confirmation processes, along with clinic contact information (email and phone).

## D. Ambiguities, Missing Information, and Assumptions

### Ambiguities:
1. **User Authentication:** No login/registration screens are shown. It's unclear if the system requires user authentication or operates as an open portal.
2. **Appointment Type Options:** The specific list of appointment types available in the dropdown is not defined beyond the services listed on the home page.
3. **Doctor Selection:** The criteria for doctor selection (specialty, availability) and the complete list of doctors are not specified.
4. **Time Slot Availability:** The mechanism for displaying only available time slots is not shown.
5. **Cancellation Process:** The confirmation or consequences of canceling an appointment are not depicted.
6. **Status Transitions:** How appointments move from "Pending" to "Confirmed" status is not explained.

### Missing Information:
1. **Error Handling:** No error messages or validation feedback screens are shown.
2. **Edit Functionality:** No ability to modify existing appointments is depicted.
3. **Past Appointments:** Only "Upcoming Appointments" are shown; no historical view is provided.
4. **Notification System:** How users receive email/SMS confirmations is not detailed.
5. **Multi-patient Support:** Whether one user can schedule for multiple patients (e.g., family members) is unclear.
6. **Appointment Duration:** No indication of how long appointments last or how this affects scheduling.
7. **Waitlist/Overflow:** No mechanism for handling fully booked time slots.

### Unclear Elements:
1. **Date Format:** The date picker shows "__/__/__" but the confirmation shows "Oct 15, 2025" - format consistency is unclear.
2. **Required vs Optional Fields:** Only "Additional Notes" is clearly optional; requirements for other fields are assumed.
3. **Dropdown Population:** How dropdowns are populated (static vs. dynamic based on availability) is not specified.
4. **Confirmation Number Generation:** The logic for generating unique confirmation numbers is not defined.

### Assumptions Made:
1. **Single User Session:** Assumed the system tracks a single patient's appointments per session.
2. **Real-time Availability:** Assumed the system checks doctor/time availability in real-time during scheduling.
3. **Immediate Confirmation:** Assumed appointments are confirmed immediately upon submission (though "Pending" status exists, suggesting possible manual review).
4. **Service-Type Alignment:** Assumed the services listed on the home page correspond to the appointment types in the dropdown.
5. **Linear Workflow:** Assumed the typical user flow is: Home → Schedule → Confirmation → My Appointments.
6. **Contact Information Persistence:** Assumed patient contact information is stored for future appointments (though no account management is shown).
7. **Cancellation Availability:** Assumed all appointments can be canceled online regardless of status or timing.
